## Goal
Reproduce the results from the DDS-SLAM paper (IROS 2024) — a neural SLAM system for deformable endoscopic surgery scenes. The paper uses the Semantic-Super and StereoMIS datasets.

## Phase 1: Repo Analysis
- Analysed the DDS-SLAM codebase (forked from IRMVLab/DDS-SLAM to bwright000/DDS-SLAM)
- Based on Co-SLAM, uses tinycudann hash grids, pytorch3d transforms, custom marching cubes C++ extension
- Original env: Python 3.7, PyTorch 1.10.1+cu113 — very outdated

## Phase 2: Environment Setup (Colab T4)
**Decision: Run on Colab T4 (16GB VRAM)** instead of local GTX 970 (4GB, insufficient).
Connected via VS Code Remote Tunnel for better dev experience than notebooks.

### Attempt 1: Exact version replication (FAILED)
Tried conda with Python 3.7 + PyTorch 1.10+cu113 to match the paper exactly.
- **Failed** because tinycudann and pytorch3d need to compile against CUDA toolkit, and Colab's system toolkit is 12.8 — can't build CUDA extensions for 11.3.
- `mathutils==2.81.2` also won't build on Python 3.10+.
- Marching cubes C++ had missing `#include <limits>` and deprecated NumPy API (`PyArray_DOUBLE`).

### Attempt 2: Modern stack with patches (SUCCEEDED)
Used Colab's native Python 3.12 + PyTorch 2.10+cu128 + CUDA 12.8 and patched the code.

**Why this is OK:** The neural network math (SDF, volume rendering, loss functions) is standard PyTorch — produces equivalent results across versions. The differences are only in deprecated API calls, not in the actual computation.

**Code patches applied:**
1. `ddsslam.py:33` — `cuda:2` → `cuda:0` (Colab single GPU)
2. `eval_ate.py` + `pose_array.py` — `mathutils` → `scipy.spatial.transform.Rotation` (quaternion conversion, careful with [w,x,y,z] vs [x,y,z,w] order)
3. `utils.py` — `plt.cm.get_cmap()` → `plt.colormaps[]` (matplotlib deprecation)
4. `ddsslam.py` — `cv2.imshow` wrapped in try/except (headless Colab)
5. `marching_cubes.cpp` — added `#include <limits>` (modern GCC)
6. `pywrapper.cpp` — `PyArray_DOUBLE/ULONG` → `NPY_DOUBLE/NPY_ULONG` (NumPy 2.x)
7. Deleted stale `_mcubes.cpp` so Cython regenerates for current NumPy
8. Added `__init__.py` to datasets/, model/, optimization/, tools/ (Python 3.12 needs explicit package markers)

**Dependencies built successfully:**
- tinycudann 2.0 (from source, TCNN_CUDA_ARCHITECTURES=75 for T4)
- pytorch3d 0.7.9 (from source)
- marching cubes (Cython + C++)

## Phase 3: Dataset Preparation
**Semantic-Super dataset** — user has raw data locally at `F:\Datasets\SemSup\v2_data\` with trials 3, 4, 8, 9.

**Critical discovery: NO depth maps in the dataset.**
- The DDS-SLAM paper says "utilize pre-trained depth estimation models provided by Semantic-SuPer to obtain deep priors"
- This means depth is a **separate preprocessing step**, not included in the data download
- The Python-SuPer repo's finetuned Monodepth2 checkpoints are on Google Drive — **inaccessible**
- The `DeepLabV3+/` folder contains semantic segmentation `.npy` files (shape 3×480×640), NOT depth maps

**No pose/ directory needed** — the code generates identity poses internally (load_poses doesn't read from disk).

## Phase 4: Depth Map Generation
Created `Addons/generate_depth.py` supporting 3 methods to compare.

### Method 1: Depth Anything V2 Metric Indoor (monocular)
- HuggingFace model, auto-downloads, simplest setup
- Outputs metric depth in meters directly
- Modern model (2024), validated on endoscopic imagery (IROS 2025)

### Method 2: Monodepth2 mono+stereo (monocular)
- Same architecture family as the paper's approach
- Auto-downloads 99MB KITTI-pretrained weights (not Google Drive)
- Outputs disparity → converted to depth via 1/disparity
- NOT finetuned on surgical data — **required manual rescaling** from driving scale (~4.5-21m) to surgical range (0.05-0.3m)

### Method 3: RAFT-Stereo (stereo) — NOT YET RUN
- Uses BOTH left and right images (closest to paper's pipeline)
- Weights from Dropbox (accessible)
- Da Vinci surgical system baseline: ~5.5mm (0.0055m)
- Formula: depth = fx × baseline / disparity ≈ 4.23 / disparity

## Phase 5: Results

### Rendering Quality Comparison (vs Paper Table I, Lab1 = trail3)

| Method | PSNR ↑ | SSIM ↑ | LPIPS ↓ | ATE RMSE |
|--------|--------|--------|---------|----------|
| **Paper (DDS-SLAM)** | **28.649** | **0.797** | **0.231** | N/A (uses Rep.Err) |
| Depth Anything V2 | 27.605 | 0.754 | 0.372 | 0.059m |
| Monodepth2 (rescaled) | 26.885 | 0.729 | 0.404 | 0.004m |

### What These Metrics Mean

**PSNR (Peak Signal-to-Noise Ratio)** ↑ higher is better
- Measures pixel-level accuracy between rendered and ground truth images
- Computed as `20 * log10(1 / sqrt(MSE))` where MSE is mean squared error
- Scale: typically 20-40 dB for images. >30 = very good, 25-30 = acceptable, <25 = poor
- Our results (~27 dB) are close to the paper (~28.6 dB) — about 1 dB gap

**SSIM (Structural Similarity Index)** ↑ higher is better
- Measures structural similarity considering luminance, contrast, and structure
- More perceptually meaningful than PSNR — captures whether images "look" similar
- Scale: 0 to 1. >0.9 = excellent, 0.7-0.9 = good, <0.7 = poor
- Our results (~0.75) vs paper (~0.80) — the structure is captured but with some blurring

**LPIPS (Learned Perceptual Image Patch Similarity)** ↓ lower is better
- Uses a pretrained neural network (AlexNet) to compare perceptual similarity
- Most aligned with human perception — captures texture quality, sharpness, fine detail
- Scale: 0 to 1. <0.1 = near identical, 0.1-0.3 = good, 0.3-0.5 = noticeable differences
- Our results (~0.37-0.40) vs paper (0.23) — biggest gap, meaning our renders are perceptually less sharp

### Analysis
- **Depth Anything V2 outperforms Monodepth2** across all metrics — metric depth matters
- **~1 PSNR gap** from paper is reasonable given we're using a different depth model
- **LPIPS gap is largest** (0.37 vs 0.23) — the paper's surgical-finetuned depth produces sharper geometric detail, leading to perceptually better renders
- **ATE is not directly comparable** — dataset has identity poses (no real ground truth camera motion), so ATE measures drift from "no movement". Monodepth2's lower ATE (0.004m) is misleading — the compressed depth range constrains poses to stay near identity
- The paper uses **Reprojection Error** (Rep.Err) as its primary tracking metric, not ATE

### Evaluation Script
Created `Addons/eval_rendering.py` — computes PSNR, SSIM, LPIPS against ground truth RGB frames. Requires `lpips` package.

## Key Learnings
1. **Don't fight the platform** — use native Colab stack, patch the few API differences
2. **Datasets often exclude depth** — many papers treat depth estimation as a preprocessing step, not part of the data download
3. **Google Drive links in papers frequently become inaccessible** — a reproducibility problem across ML research
4. **Depth scale matters hugely** — Monodepth2 KITTI-pretrained produces depths at driving scale (tens of meters). Surgical scenes are centimetres. Without rescaling, almost no pixels pass the valid depth filter, crashing training
5. **Metric vs relative depth** — DDS-SLAM uses depth in SDF loss (weight 1000) and ray sampling (±0.1m window). Metric depth (Depth Anything V2) works out of the box. Relative depth (Monodepth2) requires manual normalization
6. **Da Vinci surgical system** has a ~5.5mm stereo baseline (constrained by trocar diameter)
7. `drive.mount()` only works from Colab notebook cells, not VS Code terminal
8. Python 3.12 requires explicit `__init__.py` files for local packages
9. **Rendering metrics (PSNR/SSIM/LPIPS)** are the right comparison, not ATE — the paper uses these plus reprojection error
